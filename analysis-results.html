Reentrancy
Confidence: HIGH
Line 42 Column 12
State variables are updated after an external call, allowing malicious contracts to repeatedly call the function before the balance is safely updated.
Vulnerable Code
(bool success, ) = msg.sender.call{value: amount}(""); require(success, "Transfer failed"); balances[msg.sender] -= amount; // State is updated after the external call
How to Fix
❌ VULNERABLE CODE
1
(bool success, ) = msg.sender.call{value: amount}("");
2
require(success, "Transfer failed");
3
balances[msg.sender] -= amount; // State is updated after
✅ FIXED CODE
1
balances[msg.sender] -= amount; // Update state FIRST
2
(bool success, ) = msg.sender.call{value: amount}("");
3
require(success, "Transfer failed");
Why It Works
  • Implement the Checks-Effects-Interactions pattern.
  • Always update state variables (like balances) before making external calls to untrusted addresses.
  • Alternatively, apply a ReentrancyGuard modifier.

Unchecked Call Return Value
Confidence: MEDIUM
Line 115
The return value of a low-level external call is not checked. If the call fails, execution will continue silently, potentially causing state inconsistencies.
Vulnerable Code
targetContract.call(abi.encodeWithSignature("doSomething()"));
How to Fix
❌ VULNERABLE CODE
1
targetContract.call(abi.encodeWithSignature("doSomething()"));
✅ FIXED CODE
1
(bool success, ) = targetContract.call(abi.encodeWithSignature("doSomething()"));
2
require(success, "External call failed");
Why It Works
  • Always check the boolean return value from low-level calls (call, send, delegatecall).
  • Wrap the call in a require() statement to ensure the transaction reverts if the target contract throws an error.
